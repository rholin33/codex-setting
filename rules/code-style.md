# Codex 代码风格规则

## 核心约束

- 默认使用 ES6+ 语法；禁止 `var`。
- 默认避免 `class` 和继承；只有在框架、库约定或现有代码大量使用时，才沿用 `class`。
- 单个代码文件尽量不超过 300 行；超过时优先按功能拆分，但不为机械满足行数制造无意义抽象。
- 新增注释默认使用简体中文；英文项目或既有英文注释风格明显时，沿用项目语言。
- 以项目现有规范、格式化器和 lint 配置为准；本规则只在项目缺少明确约定时补位。

## JavaScript / TypeScript

- 变量：优先 `const`，需要重新赋值时使用 `let`。
- 函数：优先箭头函数；顶层工具函数可用具名函数声明以改善堆栈和可读性。
- 对象：优先使用属性简写、解构、可选链 `?.`、空值合并 `??`。
- 模块：优先 ES Module `import/export`；配置文件、脚本或现有 CommonJS 项目可继续使用 CommonJS。
- 异步：优先 `async/await`；并发任务按语义选择 `Promise.all` 或 `Promise.allSettled`。
- TypeScript：对象结构优先 `interface`；联合、交叉、映射等类型用 `type`；避免 `any`，必须越界时优先 `unknown` 并做类型收窄。

## 命名规范

| 场景 | 风格 | 示例 |
|------|------|------|
| 变量/函数 | camelCase | `fetchUserList` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 组件文件 | PascalCase | `UserCard.vue` |
| 组合式函数 | camelCase + use 前缀 | `usePageTitle.ts` |
| 类型/接口 | PascalCase | `UserProfile` |
| CSS 类名 | kebab-case | `.user-card` |

## 格式规范

- 优先使用项目已配置的格式化和检查命令。
- 如果项目使用 `oxfmt` 或 `oxlint`，按对应配置执行。
- 不为了个人偏好引入新的格式化器或大规模重排。

## 函数设计原则

- 单一职责：每个函数只做一件清晰的事，函数名能描述主要行为。
- 抽象层次一致：同一函数内保持相近颗粒度，细节逻辑提取为独立函数。
- 控制嵌套层级：超过 3 层嵌套时，优先用提前返回、拆分函数或数据映射简化。

## 注释规范

- 注释解释“为什么”和重要约束，不重复代码表面含义。
- 复杂公共函数、边界行为或非显然协议使用 JSDoc/TSDoc。

## 避免事项

- 避免 `var`。
- 避免无必要的 `class`、继承和深层对象层级。
- 避免裸 `console.log`；调试输出完成后删除，必要日志使用项目日志设施。
- 避免魔法数字；优先使用具名常量。
- 避免无意义的单字母变量名。
- 避免单文件过长，但不要破坏已有模块边界。
